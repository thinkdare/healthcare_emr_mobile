Design — Section 5: Facilities Wiring

FacilitiesListScreen + FacilityFormScreen already exist and work — they just need navigation entry points

What exists today

  • lib/presentation/facilities/screens/facilities_list_screen.dart — full list with popup menu (Edit / Delete), adaptive grid on wide screens, pull-to-refresh.
  • lib/presentation/facilities/screens/facility_form_screen.dart — create/edit form: name, type, address, phone, emergency access toggle.
  • Both screens use FacilityRepository directly. Both handle their own loading/error state.
  • FAB on the list screen already navigates to /facilities/add. Edit navigates to /facilities/edit.
  • Neither screen is reachable from any navigation entry point in the app.

The only change needed

In ProviderDashboardScreen drawer — the "Facilities" admin entry already shown in Section 1 calls:
Navigator.push(context, MaterialPageRoute(builder: (_) => const FacilitiesListScreen()))

In MoreScreen — same, using CupertinoPageRoute.

No new screens. No new routes. No new providers. Just two Navigator.push calls.

One small fix in FacilitiesListScreen

Currently the screen constructs its own FacilityRepository by reading ApiClient from context. This works. The only thing to verify is that ApiClient is still in the widget tree at the point the screen is pushed from the drawer — it is, since it's provided at the root main.dart level. No changes needed.

Out of scope for this sprint: The facility form's named routes (/facilities/add, /facilities/edit) are registered in main.dart. We'll leave those as-is — the list screen uses them internally for its FAB and popup menu, which continue to work fine.

Does this wiring plan look right? Confirm in terminal — that's the full design.
